Ниже представлены подробные ответы на вторую группу вопросов: **«Модели представления данных»**.

---

### 1. Опишите классификацию моделей представления данных

Модель данных — это совокупность структур данных и операций их обработки. Классификация моделей обычно строится по двум признакам: по уровню абстракции и по логической структуре.

**По уровням абстракции:**
1.  **Инфологические (концептуальные) модели:** Описывают предметную область на языке, понятном человеку (например, модель «Сущность-Связь» или ER-модель). Они не привязаны к конкретной СУБД.
2.  **Даталогические (логические) модели:** Описывают данные с точки зрения СУБД. Именно к этой категории относятся иерархическая, сетевая и реляционная модели.
3.  **Физические модели:** Описывают способы размещения данных на магнитных дисках, методы доступа и индексации.

**По логической структуре (основная классификация):**
*   **Иерархическая** (данные в виде дерева).
*   **Сетевая** (данные в виде произвольного графа).
*   **Реляционная** (данные в виде таблиц).
*   **Постреляционная и объектно-ориентированная** (современные расширения для работы со сложными объектами).
*   **NoSQL-модели** (ключ-значение, документоориентированные, графовые — для Big Data).

### 2. Опишите сетевую модель данных. Приведите достоинства и недостатки.

**Суть модели:**
Сетевая модель является расширением иерархической. Она базируется на понятии **графа**. В этой модели любой объект (запись) может быть связан с любым другим объектом. Здесь нет строгого подчинения «один ко многим»: один «потомок» может иметь несколько «предков» (владельцев).

**Основные характеристики:**
*   Связи реализуются с помощью указателей (адресов памяти).
*   Тип связи: «многие-ко-многим» (M:M).

**Достоинства:**
*   **Эффективность:** Высокая скорость доступа к данным, так как переходы осуществляются по прямым физическим адресам (указателям).
*   **Гибкость:** Возможность моделировать любые сложные взаимосвязи между объектами без дублирования данных.

**Недостатки:**
*   **Сложность:** Очень сложная структура и логика навигации. Программист должен точно знать «путь» к данным.
*   **Жесткость:** Изменение структуры базы (например, добавление нового типа связи) требует переписывания прикладных программ.
*   **Трудность контроля:** Сложно обеспечить целостность данных из-за обилия перекрестных ссылок.

### 3. Опишите иерархическую модель данных. Приведите достоинства и недостатки.

**Суть модели:**
Данные представляются в виде древовидной структуры. Имеется один корень (верхний уровень), от которого расходятся ветви к подуровням.

**Основные характеристики:**
*   Связь типа «один-ко-многим» (1:N): один родитель может иметь много потомков, но у каждого потомка — строго один родитель.
*   Доступ к любому элементу возможен только через его предка (сверху вниз).

**Достоинства:**
*   **Скорость:** Очень высокая производительность при запросах, которые соответствуют иерархии (например, «найти всех сотрудников отдела X»).
*   **Простота концепции:** Легко визуализировать (как файловую систему или организационную структуру компании).
*   **Целостность:** Автоматическое поддержание иерархической логики (при удалении родителя удаляются все потомки).

**Недостатки:**
*   **Ограниченность:** Невозможно напрямую реализовать связь «многие-ко-многим».
*   **Дублирование:** Чтобы связать ребенка с двумя родителями, приходится дублировать данные ребенка в разных ветвях.
*   **Асимметрия поиска:** Поиск «снизу вверх» (от потомка к родителю) или поиск по критериям, не входящим в иерархию, выполняется крайне медленно.

### 4. Опишите реляционную модель данных. Приведите достоинства и недостатки.

**Суть модели:**
Разработана Эдгаром Коддом в 1970 году. Основана на математическом понятии «отношение» (relation). Данные представляются в виде совокупности двумерных **таблиц**.

**Основные характеристики:**
*   Данные в таблицах логически связаны через значения ключевых полей.
*   Отсутствуют видимые пользователю физические указатели.
*   Строгий математический фундамент (реляционная алгебра).

**Достоинства:**
*   **Простота и доступность:** Табличная форма понятна любому пользователю.
*   **Независимость данных:** Логическое представление данных отделено от способов их хранения. Изменение структуры таблицы часто не требует изменения программ.
*   **Гибкость запросов:** Благодаря языку SQL можно извлекать данные в любых сочетаниях, которые не были предусмотрены при проектировании.
*   **Целостность:** Развитые механизмы контроля (первичные и внешние ключи).

**Недостатки:**
*   **Производительность:** Требует значительных ресурсов CPU и памяти для выполнения операций соединения (JOIN) нескольких таблиц.
*   **Семантический разрыв:** Трудности при отображении сложных иерархических или сетевых объектов (объектно-реляционный разрыв).
*   **Ограничения масштабируемости:** Классические реляционные БД сложно масштабировать горизонтально (на множество серверов) по сравнению с NoSQL решениями.
